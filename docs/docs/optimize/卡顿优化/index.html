    卡顿优化 #

    照成卡顿的原因由很多种，最终都会反映到cpu时间上，cpu时间分为用户时间，系统时间。

    卡顿优化的手段 #

    https://github.com/Tencent/matrix/wiki/Matrix-Android-TraceCanary

    Android的两种Java层面监控的方法： #

    • Looper
    public static void loop() {
  //...
    for (;;) {
        //...
        // This must be in a local variable, in case a UI event sets the logger
        final Printer logging = me.mLogging;
        if (logging != null) {
            logging.println(">>>>> Dispatching to " + msg.target + " " +
                    msg.callback + ": " + msg.what);
        }
        //...
          msg.target.dispatchMessage(msg);
        //
        if (logging != null) {
            logging.println("<<<<< Finished to " + msg.target + " " + msg.callback);
        }
        //...
    }
}

public void setMessageLogging(@Nullable Printer printer) {
    mLogging = printer;
}

    由于每个message都会在dispatchMessage这一句中分发完成，而logging会在前后各打一次log，所以其实我们就可以知道这一次message的执行超时了。 在之外的地方我们可以通过一个另外的线程按照一定的频率获取主线程堆栈信息，当发生有message的处理超时的情况时就可以保存这一些堆栈信息并上报。

    • Choreographer
    Choreographer.getInstance().postFrameCallback(new Choreographer.FrameCallback() {
    @Override    
    public void doFrame(long frameTimeNanos) {
        if(frameTimeNanos - mLastFrameNanos > 100) {
            ...
        }
        mLastFrameNanos = frameTimeNanos;
        Choreographer.getInstance().postFrameCallback(this);
    }
});

    当两次 vsync的间隔超出一定的时间就意味着发生了卡顿，这时候进行堆栈信息的保存，会有点太晚了。

    检测到卡顿之后如何获取现场呢？ #

    • 一个后台线程按照一定频率来保存主线程的堆栈信息，然后当发生了卡顿之后将堆栈保存下来。
    • 发送sigquit方法模拟anr ，来生成traces.txt文件的生成。
    • 预埋点，或者现加埋点 使用Debug.startMethodTracing来生成trace文件 // - 字节码插桩，所有函数都加上
